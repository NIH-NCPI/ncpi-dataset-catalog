# Variable Classification Pipeline

Classifies ~425K dbGaP phenotype variables into ~6,700 searchable concepts
organized in a 4-level hierarchy. 179K variables (42%) are currently classified.

## Prerequisites

```bash
# Python venv (shared with the backend)
cd backend && python -m venv .venv && .venv/bin/pip install -r requirements.txt

# Anthropic API key (for LLM steps)
export ANTHROPIC_API_KEY='sk-ant-...'
# or add to backend/.env
```

## Quick start

```bash
cd catalog-build/classification

# See available targets
make help

# View the full concept hierarchy with variable counts
make hierarchy

# Rebuild the DuckDB search index after any data changes
cd ../../backend && make db-reload
```

## Pipeline overview

The classification pipeline runs in stages. Each stage is incremental and
resumable — results are cached per-study or per-concept.

### Stage 1: Parse variable metadata

dbGaP publishes `var_report.xml` files for each study. The parser extracts
variable names, descriptions, and PHV IDs, deduplicating across consent groups.

```bash
# This is handled by the catalog build scripts in catalog-build/
npm run update-dbgap-source
```

### Stage 2: Classify variables (Haiku)

Each variable is classified by Claude Haiku into a canonical concept based on
its name and description. Variables are batched by study and table.

```bash
make classify                              # All studies (resumable)
make classify-study STUDY=phs000007        # Single study
```

Output: `output/llm-concepts-v4/*.json` (one file per study)

Cost: ~$250 for a full run of all ~2,944 studies.

### Stage 3: Build vocabulary and hierarchy

The concept vocabulary comes from three sources:

| Source | Script | Concepts | Namespace |
|--------|--------|----------|-----------|
| TOPMed harmonized variables | `extract_topmed_seeds.py` | ~360 | `topmed:` |
| PhenX Toolkit protocols | `build_phenx_vocabulary.py` | ~210 | `phenx:` |
| Vocabulary expansion (LLM) | `expand_vocabulary.py` | ~130 | `topmed:` / `phenx:` |

The ISA hierarchy (`concept-isa.json`) defines parent-child relationships
between concepts. It is generated by LLM and manually curated.

```bash
make seeds          # Extract TOPMed seed concepts
make phenx-vocab    # Build PhenX vocabulary
make isa            # Generate ISA hierarchy (LLM)
make phenx-inject   # Inject PhenX ground-truth mappings
make expand         # Expand vocabulary with new categories (LLM, ~$2)
```

### Stage 4: Generate archetypes (Sonnet)

Concepts with >200 variables get broken into "archetypes" — canonical
measurement patterns that group semantically identical variables. For example,
`topmed:ecg` (10,541 vars) becomes 41 archetypes like
`ncpi:ecg_atrial_fibrillation` (136 vars) and `ncpi:ecg_qt_interval` (114
vars).

```bash
make archetypes                                    # All eligible concepts (~$15-20)
make archetypes-concept CONCEPT=topmed:ecg         # Single concept
make archetypes-dry-run                            # Stats only, no LLM calls
```

Output:
- `output/archetypes/*.json` — cached LLM results per concept
- `output/concept-vocabulary.json` — updated with archetype entries
- `output/concept-isa.json` — updated with archetype ISA edges
- `output/llm-concepts-v4/*.json` — variables re-tagged to archetypes

Current stats: 6,281 archetypes across 185 concepts, 98K variables assigned.

### Stage 5: Rebuild search index

After any classification or vocabulary changes, rebuild the DuckDB index:

```bash
cd ../../backend
make db-reload    # Rebuilds DuckDB + restarts API server
```

## Full rebuild from scratch

```bash
cd catalog-build/classification

# 1. Classify all variables (~$250, several hours)
make classify

# 2. Build vocabulary and hierarchy
make v4              # namespace + phenx-vocab + isa + phenx-inject

# 3. Expand vocabulary with additional categories (~$2)
make expand

# 4. Generate archetypes for large concepts (~$15-20, ~30 min)
make archetypes

# 5. Rebuild search index
cd ../../backend && make db-reload
```

## Output files

| File | Description |
|------|-------------|
| `output/llm-concepts-v4/*.json` | Per-study classification (one file per study) |
| `output/concept-vocabulary.json` | All concepts with names, descriptions, CUIs |
| `output/concept-isa.json` | ISA hierarchy edges (child → parent) |
| `output/ncpi-categories.json` | Top-level category definitions |
| `output/archetypes/*.json` | Cached archetype LLM results per concept |

## Hierarchy structure

```
Category (27)                          ← top-level grouping
  └─ Concept (~580)                    ← TOPMed/PhenX concept
       └─ Sub-concept (optional)       ← further grouping
            └─ Archetype (~6,300)      ← leaf node, groups identical variables
```

Max depth: 4. ISA closure means querying any node returns all descendant
variables. The resolve agent navigates the tree to find the right level of
specificity for each query.
